00

O que o Managed Agents resolve na prática

A grande mudança aqui não é "mais um SDK". É a Anthropic assumir o que normalmente trava um projeto de agentes antes de ele virar produto.

Antes

  • Você precisava montar o loop do agente sozinho.
  • Tinha que decidir como lidar com contexto, retomada, tool use e falhas.
  • Precisava provisionar sandbox, rede, arquivos e observabilidade.
  • Deploy significava costurar aplicação, infraestrutura e segurança ao mesmo tempo.

Agora

  • O agente roda em uma harness gerenciada pela Anthropic.
  • A plataforma cuida de sandbox, compaction, caching e execução longa.
  • Você trabalha com interfaces estáveis: agente, ambiente, sessão e eventos.
  • Seu foco vai para caso de uso, ferramentas, entradas, saídas e produto.

Leitura correta: Managed Agents é melhor quando o trabalho é assíncrono, de maior duração e com cara de "pega esse objetivo e vai até o fim". Para loops super customizados e controle fino, a Anthropic continua posicionando a Messages API como a rota certa.

01

O setup mínimo para começar hoje

Se você quer sair do conteúdo e chegar na execução, esta é a linha de montagem mais curta.

1. Gere sua API key

A chave sai no Claude Console. Separe workspaces por caso de uso e não exponha a key no frontend.

2. Instale o CLI

Para macOS, a própria documentação recomenda começar com o ant via Homebrew.

3. Escolha um job simples

Exemplo bom: benchmark, revisão de conteúdo, análise de docs, triagem de repo ou QA operacional.

Instale e valide o CLI

brew install anthropics/tap/ant ant --version

Se você não estiver no macOS, use o instalador equivalente da página oficial do CLI da Anthropic.

Configure a variável de ambiente

echo 'export ANTHROPIC_API_KEY=sk-ant-api03-...' >> ~/.zshrc source ~/.zshrc

O ant lê a chave a partir de ANTHROPIC_API_KEY. Se preferir SDK, o mesmo nome de variável continua valendo.

Ponto importante: os recursos de Managed Agents ainda usam beta versionada. No CLI isso já vem resolvido automaticamente; em HTTP cru, você precisa mandar anthropic-beta: managed-agents-2026-04-01.

02

Defina o agente como código

O jeito mais limpo de não transformar seu agente em gambiarra é tratá-lo como artefato versionado no repositório.

Arquivo: first-agent.agent.yaml

name: First Production Agent model: claude-sonnet-4-6 system: | Você é um agente operacional. Receba um objetivo claro, execute o trabalho até o fim, use ferramentas quando precisar e salve a entrega final em /mnt/session/outputs/. tools: - type: agent_toolset_20260401 default_config: enabled: true permission_policy: type: always_allow

O agent_toolset_20260401 habilita o conjunto pronto de ferramentas do Managed Agents: bash, operações de arquivo, web search e outras primitivas de execução. Para um primeiro fluxo automatizado, essa configuração já te coloca em jogo.

Quando usar esse formato

  • Quando você quer iterar rápido e versionar tudo em YAML.
  • Quando vai atualizar o agente em CI/CD depois.
  • Quando precisa de uma base reaproveitável em múltiplas sessões.

Cuidado com permissões

  • always_allow faz sentido em sandbox controlado e caso de uso bem delimitado.
  • Se o job tocar sistemas críticos, endureça a política antes do deploy real.
  • Primeiro agente em produção não significa abrir tudo sem critério.
03

Defina o ambiente onde ele vai trabalhar

O ambiente é o template do container. É aqui que você escolhe rede, runtime e o grau de liberdade do agente.

Arquivo: first-agent.environment.yaml

name: first-production-env config: type: cloud networking: type: unrestricted

Para o primeiro agente, cloud + unrestricted deixa o caminho mais simples, especialmente se você quer usar web search e benchmarks externos. Depois você pode travar isso.

Use unrestricted quando

  • O agente precisa consultar web, documentação externa ou endpoints públicos.
  • Você está em protótipo ou em um job de pesquisa com baixo risco operacional.
  • O objetivo é reduzir fricção no primeiro deploy.

Migre para limited quando

  • O fluxo já estiver maduro e não depender de saída livre para internet.
  • Você quiser reduzir superfície de risco no ambiente.
  • O agente operar só com arquivos, GitHub montado ou ferramentas específicas.
04

Crie os recursos e rode a primeira sessão

Aqui é onde a teoria vira agente rodando. Você vai criar o agente, o ambiente, abrir uma sessão e mandar o primeiro job.

  1. Crie o agente

    O CLI devolve um agent.id. Guarde isso; ele é o identificador que seu backend vai usar depois.

  2. Crie o ambiente

    O ambiente também recebe um ID próprio. Essa separação é o que te permite reaproveitar a mesma configuração em várias sessões.

  3. Abra a sessão

    A sessão é a run concreta do trabalho. Pense nela como o job que liga agente e ambiente para executar uma tarefa específica.

  4. Envie o objetivo

    O segredo aqui é pedir algo delimitado, com saída clara e critério de término explícito.

  5. Acompanhe até ficar idle

    O sinal de término é a sessão ir para idle. A partir daí, você decide se retorna resultado, armazena artefato ou dispara outro passo.

AGENT_ID=$(ant beta:agents create --transform id --format yaml < first-agent.agent.yaml) ENV_ID=$(ant beta:environments create --transform id --format yaml < first-agent.environment.yaml) SESSION_ID=$(ant beta:sessions create \ --agent "$AGENT_ID" \ --environment "$ENV_ID" \ --title "Primeiro agente em produção" \ --transform id --format yaml)
ant beta:sessions:events send \ --session-id "$SESSION_ID" \ --event '{type: user.message, content: [{type: text, text: "Pesquise três concorrentes de CRM para PMEs no Brasil, escreva um resumo comparativo em Markdown e salve o resultado em /mnt/session/outputs/benchmark.md."}]}' ant beta:sessions stream --session-id "$SESSION_ID"

Boa prática: peça sempre uma saída verificável. "Salve em /mnt/session/outputs/..." ou "pare quando o arquivo existir" reduz muito a chance do agente encerrar sem entrega concreta.

05

Conecte dados reais sem entregar suas credenciais ao agente

O primeiro deploy fica muito melhor quando você liga o agente em inputs de verdade. A boa notícia é que a plataforma já foi desenhada para isso.

Arquivos

PDFs, CSVs, briefings e anexos entram via Files API e podem ser montados na sessão como recurso de trabalho.

GitHub

Você pode montar um repositório no container da sessão e conectar o GitHub MCP para abrir PRs e trabalhar no repo.

MCP e sistemas externos

A Anthropic posiciona MCP como a rota certa para integrar ferramentas e manter autenticação fora do sandbox do agente.

Por que isso importa

No post de engenharia da Anthropic, a ideia central é separar o "cérebro" das "mãos". Isso evita que o sandbox onde o agente executa código tenha acesso direto aos tokens e segredos da sua operação.

Tradução prática

  • Não enfie credencial no prompt.
  • Não trate o sandbox como lugar para segredos.
  • Use recursos de sessão, GitHub MCP e vaults externos como desenho padrão.
06

Como isso vira deploy de verdade

Produção não é "o agente funcionou uma vez". Produção é conseguir atualizar, observar e reaproveitar esse agente sem depender de ritual manual.

O que fica com a Anthropic

  • Harness do agente
  • Sandbox e runtime gerenciado
  • Timeline de sessão e eventos
  • Abstração de agent, environment e session

O que continua sendo seu

  • Qual job dispara o agente
  • Quais entradas e saídas viram produto
  • Guardrails, custo, rate limiting e downstream actions
  • Persistência do resultado para o seu app, CRM, dashboard ou fila
  1. Versione os YAMLs no repositório

    *.agent.yaml e *.environment.yaml viram parte do seu deploy, não algo escondido no histórico do terminal.

  2. Sincronize esses recursos no CI/CD

    A própria documentação do CLI recomenda criar e atualizar agentes e ambientes como arquivos versionados.

  3. Seu backend cria uma sessão por job

    Quando um usuário pede algo no produto, seu backend cria a sessão, envia o evento e acompanha o status.

  4. Escolha stream ou polling

    Stream é ótimo para UX ao vivo. Polling é melhor para jobs de background e workers mais robustos.

  5. Persista o resultado fora da sessão

    O agente conclui o trabalho; seu produto precisa decidir onde isso vive depois: banco, storage, CMS, GitHub, Slack ou CRM.

AGENT_VERSION=$(ant beta:agents retrieve \ --agent-id "$AGENT_ID" \ --transform version --format yaml) ant beta:agents update \ --agent-id "$AGENT_ID" \ --version "$AGENT_VERSION" \ < first-agent.agent.yaml ant beta:environments update \ --environment-id "$ENV_ID" \ < first-agent.environment.yaml

Resumo honesto: Managed Agents te poupa de construir a infraestrutura do agente, mas não substitui o desenho do produto. O deploy real acontece quando esse agente entra no seu fluxo operacional com gatilho, observabilidade e destino de resultado.

07

Observabilidade: como saber se o agente travou, terminou ou queimou custo

O erro mais comum em agente "em produção" é rodar sem visibilidade. A Anthropic já te entrega dois pontos fortes aqui.

Claude Console

A timeline mostra lista de sessões, status, eventos cronológicos, uso de tokens e tool execution.

Eventos brutos

Você pode listar eventos da sessão programaticamente para debugar erros, entender tool calls e medir custo.

Stop reason

Para automação séria, trate idle e end_turn como parte do protocolo do seu worker.

ant beta:sessions:events list \ --session-id "$SESSION_ID" \ --format explore
Se a sessão parou cedo Abra os eventos e veja o último tool_use, erro de sessão ou status de parada.
Se o custo subiu Use timeline e eventos para identificar sessões longas, prompts frouxos e loops desnecessários.
Se o agente "terminou" sem entregar Volte ao prompt e explicite formato de saída, caminho de arquivo e condição de encerramento.
Se você precisa de auditoria Persista ID de sessão, timestamps e status final junto ao registro do job no seu sistema.
08

Quando usar Managed Agents e quando não usar

Isso evita a expectativa errada logo no início.

Use Managed Agents Quando o trabalho é longo, assíncrono, precisa de ferramentas e você quer a infraestrutura do agente já gerenciada.
Use Messages API Quando você quer construir sua própria harness, controlar cada passo e operar com loop customizado.
Primeiro caso de uso ideal Pesquisa operacional, análise de documentos, benchmark, revisão de código, triagem e jobs com começo, meio e fim claros.
Primeiro caso de uso ruim Algo amplo demais, sem saída definida, com permissões críticas e sem critério objetivo de conclusão.
