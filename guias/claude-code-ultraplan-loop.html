01

O que ULTRAPLAN parece ser

O nome importa menos do que o conceito. A leitura mais util e que a Anthropic parece ter explorado uma camada onde o agente recebe tempo e ambiente para pensar profundamente antes de agir.

Planning offloaded

Em vez de gastar todo o raciocinio no mesmo loop da execucao, o sistema separaria um bloco de planejamento com mais folego e mais custo computacional.

Interface de aprovacao

A ideia central faz sentido em tarefas caras de errar: primeiro revisar o plano, depois liberar a execucao.

Cloud first

Esse tipo de planning profundo combina mais com sessao remota ou web do que com uma interacao curta de terminal.

Nao e doc publica

Em 2 de abril de 2026, eu nao encontrei documentacao oficial aberta da Anthropic usando o nome ULTRAPLAN como feature publica.

Sinal de produto

Mesmo sem ship oficial, o conceito mostra uma direcao clara: planning vira produto, nao so fase informal da conversa.

Licao pratica

Para builders, a grande licao e separar estrategia de execucao quando o custo de errar na abordagem e alto.

02

O que existe publicamente hoje no lugar disso

Mesmo sem ULTRAPLAN publico, a stack atual ja te deixa montar um fluxo muito parecido com planejamento, aprovacao e execucao.

Plan mode

  • Claude analisa seu codigo e cria um plano sem modificar arquivos ou rodar comandos.
  • Bom para tarefas complexas em que voce quer revisar a estrategia primeiro.
  • Depois de aprovar, voce pode trocar para modos de execucao como Ask permissions ou Auto accept edits.

Remote execution

  • claude --remote cria uma sessao web rodando em cloud.
  • No Desktop, voce tambem pode escolher ambientes Remote para execucao cloud.
  • Isso ja te permite planejar e depois mandar executar em outra superficie ou infraestrutura.
# planejar primeiro
claude --permission-mode plan

# depois mandar executar remotamente
claude --remote "Execute o plano em docs/migration-plan.md"

Leitura pratica: voce nao precisa esperar um "ULTRAPLAN" publico para operar com esse modelo mental. O fluxo plan first, then run ja esta disponivel.

03

Como a recorrencia funciona de verdade hoje

A melhor ancoragem publica para o conceito de /loop hoje e a camada de scheduled tasks no Claude Code Desktop.

Onde existe No Claude Code Desktop, com tarefas locais e remotas agendadas.
Frequencias Manual, hourly, daily, weekdays e weekly. Para intervalos fora do picker, a doc diz que voce pode pedir em linguagem natural.
Como roda local O app checa o schedule a cada minuto enquanto esta aberto e cria uma sessao nova quando a hora chega.
Se o computador dorme Runs locais podem ser perdidos. Na volta, o Desktop tenta um catch-up da execucao mais recente perdida.
Se voce quer rodar mesmo off A doc recomenda remote tasks para tarefas que precisam continuar mesmo quando sua maquina esta desligada.
Permissoes Cada task tem seu proprio permission mode. Em Ask mode, a run pode ficar esperando aprovacao.
04

O modelo certo para automacao de longo prazo

O ganho real aqui nao e "deixar uma tarefa rodando para sempre". E montar uma esteira com checkpoints certos.

1. Planejar primeiro

Use plan mode para desenhar a estrategia quando a tarefa for cara de errar ou envolver muitas dependencias.

2. Aprovar a abordagem

Revise o plano antes de automatizar a recorrencia. O erro mais caro e repetir a estrategia errada em loop.

3. Escolher o ambiente

Local se voce precisa do seu filesystem e setup. Remote se quer resiliencia e execucao sem depender da sua maquina acordada.

4. Agendar com guardrails

Toda automacao recorrente precisa de instrucoes do tipo "so rode se..." ou "se passar do horario, faca X". A doc recomenda esse cuidado explicitamente.

5. Revisar permissoes

Execute uma vez manualmente, aprove sempre allow onde faz sentido e evite travas silenciosas depois.

6. Tratar como sistema

Planejamento, execucao e recorrencia sao camadas diferentes. Quando voce separa isso bem, a automacao fica muito mais confiavel.

05

O que tratar com cautela

Essa parte protege seu conteudo de soar como anuncio oficial de algo que ainda parece interno.

Pode afirmar com seguranca

  • O leak sugere uma direcao de planejamento mais pesado e separado da execucao.
  • Plan mode, remote tasks e scheduled tasks ja existem publicamente no ecossistema Claude Code.
  • Scheduled tasks no Desktop ja permitem recorrencia real com frequencias, permissoes e historico.
  • Remote tasks sao a opcao oficial quando voce quer rodar mesmo com sua maquina desligada.

Melhor nao vender como fato final

  • Dizer que ULTRAPLAN ja e recurso publico com esse nome nas docs abertas.
  • Tratar /loop como unica ou principal forma oficial de recorrencia se as docs publicas ancoram isso em scheduled tasks.
  • Prometer automacao "infinita" sem mencionar ambiente, permissoes, sleeps e missed runs.
  • Assumir que planejamento em cloud e aprovacao explicita ja existem exatamente como no leak.
